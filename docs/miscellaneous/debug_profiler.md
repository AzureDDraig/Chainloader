# Debug Profiler: Logging & Class Transformation Tracing

ChainLoader is a complex bytecode-level engine. Diagnosing conflicts, load errors, or verify errors requires clear logging and tracing systems. 

ChainLoader contains a custom logger (`Logging.java`) that handles stream redirection and a class dumping mechanism inside the classloader (`ChainClassLoader.java`) to output transformed classes for inspection.

---

## 1. Redirecting stdout & stderr Streams

Standard game output (`System.out`) and error output (`System.err`) contain logs from the game engine, third-party libraries, and mods. To prevent console spam while preserving details, ChainLoader redirects these streams on startup.

### Stream Splitter (Logging.java)
```java
public static void initialize() {
    try {
        Path logsDir = Paths.get("logs");
        if (!Files.exists(logsDir)) {
            Files.createDirectories(logsDir);
        }

        originalOut = System.out;
        originalErr = System.err;

        // Open file logs
        stdoutLogStream = new FileOutputStream(new File("logs/stdout.log"), false);
        stderrLogStream = new FileOutputStream(new File("logs/stderr.log"), false);
        debugLogWriter = new PrintWriter(new FileWriter("logs/debug.log", false), true);

        // Redirect System.out to both original console and stdout.log
        System.setOut(new PrintStream(new OutputStream() {
            @Override
            public void write(int b) throws IOException {
                originalOut.write(b);
                stdoutLogStream.write(b);
            }
            @Override
            public void write(byte[] b, int off, int len) throws IOException {
                originalOut.write(b, off, len);
                stdoutLogStream.write(b, off, len);
            }
            @Override
            public void flush() throws IOException {
                originalOut.flush();
                stdoutLogStream.flush();
            }
        }, true));

        // Redirect System.err to both original console and stderr.log
        System.setErr(new PrintStream(new OutputStream() {
            @Override
            public void write(int b) throws IOException {
                originalErr.write(b);
                stderrLogStream.write(b);
            }
            @Override
            public void write(byte[] b, int off, int len) throws IOException {
                originalErr.write(b, off, len);
                stderrLogStream.write(b, off, len);
            }
            @Override
            public void flush() throws IOException {
                originalErr.flush();
                stderrLogStream.flush();
            }
        }, true));
        
        // Register shutdown hook to safely close log file streams
        Runtime.getRuntime().addShutdownHook(new Thread(() -> {
            try {
                if (stdoutLogStream != null) stdoutLogStream.close();
                if (stderrLogStream != null) stderrLogStream.close();
                if (debugLogWriter != null) debugLogWriter.close();
            } catch (Exception ignored) {}
        }));
    } catch (Exception e) {
        System.err.println("Failed to initialize log redirection: " + e.getMessage());
    }
}
```

## 2. Debug Log Level

The `logs/debug.log` file gathers verbose information about:
*   Mod JAR scanning and dependency checks.
*   Class transformation stages (annotation stripping, redirects, access widening).
*   Event translation posting.

This is enabled by passing `-Dchainloader.debug=true` or setting the `CHAINLOADER_DEBUG` environment variable to `true`.

```java
private static final boolean DEBUG = Boolean.getBoolean("chainloader.debug") || "true".equalsIgnoreCase(System.getenv("CHAINLOADER_DEBUG"));
```

If `DEBUG` is true, messages are sent to both the console and the `debug.log` file. If false, messages are only written to the `debug.log` file, keeping the console output clean.

## 3. Bytecode Dump Tracing (Class Dumping)

When troubleshooting JVM `VerifyError` crashes (which occur if generated bytecode contains invalid frames, stack maps, or type mismatches), looking at text logs is often insufficient. Developers need to analyze the generated `.class` files using decompilers.

Inside `ChainClassLoader.findClass`, ChainLoader intercepts the class loading flow for specific core classes and dumps the transformed byte array to the project's root folder:

```java
// ChainClassLoader.java - findClass Dump Hook
if (name.equals("net.minecraftforge.registries.ForgeRegistries$Keys") || 
    name.equals("net.minecraftforge.registries.ForgeRegistries") ||
    name.equals("net.chainloader.loader.core.gui.MainMenuHelper")) {
    try {
        // Writes bytecode to root: e.g. "transformed_net.minecraftforge.registries.ForgeRegistries_Keys.class"
        java.nio.file.Files.write(
            java.nio.file.Paths.get("transformed_" + name.replace('$', '_') + ".class"), 
            transformedBytes
        );
        System.out.println("[ChainClassLoader] Dumped transformed class bytes: " + name);
    } catch (Exception e) {
        e.printStackTrace();
    }
}
```

This bytecode dump can then be analyzed using standard disassemblers (like `javap` or bytecode tools) to inspect the injected instructions, verify that frames are computed correctly, and check that references map to active runtime targets.
